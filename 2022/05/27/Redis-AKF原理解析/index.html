Redis AKF原理解析

2022-05-27 Redis AKF, Redis 评论

单节点Redis有什么问题

单节点Redis搭建运维都很方便，无需考虑节点间的一致性问题。但是单节点Redis受制只有一台Redis节点，也会有一些问题

  • 单点故障：因为是单点的，当这个节点挂掉之后，Redis就不可用，对于业务会造成影响
  • 容量：单点Redis容量是有限，而且Redis是内存型的，也不能无限扩展单节点的内存，对于Redis内存管理，维护也是一个挑战
  • 压力：单点Redis性能有限，连接压力有限

什么是AKF

鉴于单节点Redis存在以上三个问题，可以通过AKF理论来解决上述的问题。AKF可扩展立方体(Scalability Cube)是从X、Y和Z轴上来指导如何进行分布式系统的架构和划分。

X轴

基于X轴做水平扩展，建立多个节点，构建成主从模式。其中主节点承担读写，从节点承担读，实现读写分离。每个节点存放的是全量数据。能够解决单点故障和一部分的压力。

Y轴

基于Y轴按照功能、业务进行划分，不同业务的数据存储到不同Redis节点。比如用户信息存到一个Redis节点，订单信息存到另一个Redis节点，能够解决容量和压力。

Z轴

基于Z轴数据分区，Y轴上按照业务划分之后，比如用户独立一个Redis，也可能遇到容量比较大，一个Redis还是放不下，这时候可以按照Z轴对数据进行分区，进一步拆分到不同的Redis节点中。

主从一致性

当按照AKF的指导原则来构建分布式系统时，又会遇到一个新的问题，那就是这么多Redis节点之间如何做数据同步，怎么保证一致性呢？

强一致性

客户端写入主节点之后，主节点会以同步阻塞的方式写入从节点，此时就是强一致性。但是会破坏可用性，当某个节点无法写入时，通过同步阻塞就会严重影响可用性。

弱一致性

客户端写入主节点之后，主节点会以异步的方式写入从节点，此时就是弱一致性，可能会丢失数据。

最终一致性

一段时间内可能是不一致的，但是保证最终是一致的。客户端写入主节点之后，主节点同步写入Kafka(kafka本身保证可靠性)，然后从节点消费Kafka来完成数据同步。

主节点的故障转移

不管是主从，还是主备，都是为了保证主的高可用性(HA)，当主节点故障时怎么完成故障的转移，以及怎么判断主节点发生故障呢？

如何判定故障

为了判断主节点是否健康，可能需要另外一部分监控节点，来负责监控主节点的监控。

强一致

需要所有监控节点都给出OK，但是在分布式系统中，依赖全部都监控节点本身就不太可靠，比如可能发生网络分区等，无法收集到全部监控节点的投票。

过半

只要超过一半认为主节点挂掉，那么即可认为主节点挂掉，此时即使剩下一批都认为主节点还是健康，但是无法超过半数，不影响结果。

人工切换

通过Redis本身提供的主从控制命令，来完成主节点的切换。

seatinel

seatinel哨兵模式可以完成对主节点的故障的自动切换，能够支撑多套Redis环境，也就是通过一套seatinel来实现Redis Y轴上的扩展。
